# Architecture Decision Records (ADR)

This directory holds law-nexus Architecture Decision Records (ADRs) in the
**MADR** (Markdown Any Decision Records) format. An ADR is a durable,
review-driven record of a consequential architectural decision: the context,
the decision, the alternatives considered, and the consequences.

## What an ADR is for

**Reader:** an engineer or agent who needs to understand why the architecture is
shaped the way it is, or who is about to make a change that touches an
architectural boundary.

**Post-read action:** decide whether a decision warrants an ADR; if it does,
write one that follows the MADR template, the D098 lifecycle tags, and the split
with GSD decisions described below.

ADRs answer **how the system is built** and **what was decided**. They are
architectural substance, not governance events.

## When to write an ADR — significance gating

Not every decision becomes an ADR. ADRs are reserved for decisions that are
consequential, hard to reverse, and likely to influence future work. Write an
ADR when a decision meets **any** of these criteria:

| Criterion | Trigger |
|---|---|
| **Architectural** | Changes the structure of the system — a new layer, a boundary move, a foundational component, or the shape of a major subsystem. |
| **Layer / contract change** | Alters a contract between layers, modules, or teams: an interface, a data model, a serialization format, or an API shape that others depend on. |
| **New port / adapter** | Introduces a new external integration point, adapter, provider binding, or source-of-truth boundary. |
| **Persistent tech choice** | Adopts, replaces, or freezes a technology, library, framework, or runtime that is expensive to change later. |

If a decision does not meet any criterion, do **not** create an ADR. Capture it
as a short note in `.gsd/DECISIONS.md`, a commit message, or inline
documentation instead. Over-granular ADRs dilute the record and bury the
decisions that matter.

**Cluster rule:** decisions that move together should be documented together.
Do not create one ADR per micro-choice when the real decision is a coupled
architectural cluster.

**Check before creating:** scan the index below for conflicts or supersession.
A new ADR that contradicts an active one must declare the relationship
explicitly (`supersedes`, `conflicts`, `amends`).

> **Origin:** the significance-gating discipline and the deterministic-template
> discipline below are adapted (not copied verbatim) from the spec-kit-plus ADR
> mechanics research. The expanded significance dimensions (impact, scope,
> proof burden, supersession risk, agent-drift risk) inform the gate; the
> scaffold-then-fill-then-validate creation sequence informs the workflow.

## Relationship to GSD decisions (`.gsd/DECISIONS.md`)

ADR and GSD decisions are **complementary, not duplicate**. They have different
owners, purposes, and lifecycles. This split is intentional and must be
preserved.

| | `doc/adr/` (this directory) | `.gsd/DECISIONS.md` |
|---|---|---|
| **Captures** | **What** was decided and **how** the architecture is built — context, decision, alternatives, consequences. | **When** and **why** a decision was made within the GSD workflow — milestone/slice links, revisit triggers, decision authority. |
| **Owner** | Architecture (human-authored, review-driven). | GSD governance (auto-generated by `gsd_decision_save`). |
| **Format** | MADR Markdown file, one decision per file. | Single table, one row per decision (`D0xx`). |
| **Lifecycle** | Status field **plus** a mandatory D098 lifecycle tag (see below). | Status and revisit triggers tracked in the GSD DB. |
| **Audience** | Engineers and agents reasoning about the architecture. | The GSD workflow tracking decision provenance and revisit points. |

**The split, concretely:**

- An **ADR** describes the architectural decision itself — the substance.
- A **D0xx row** describes the governance event around it — when it happened,
  what triggered it, and when to revisit.
- They may **reference** each other: an ADR links to its `D0xx` row for
  governance context; a `D0xx` row may point to an ADR for architectural detail.

GSD decisions (`D0xx`) are **available as preloaded context** when writing an
ADR — they inform the Context section — but an ADR is **not generated from** a
`D0xx` row. An ADR is an independent architectural record written for a cold
reader. Do not copy `D0xx` prose into an ADR verbatim; re-state the
architectural substance in the ADR's own terms.

## Numbering

ADRs are numbered `ADR-NNNN`, zero-padded to four digits: `ADR-0001`,
`ADR-0002`, … `ADR-0042`.

- Numbers are **monotonic and never reused**. A superseded ADR keeps its number;
  its successor takes the next free number.
- The filename is `ADR-NNNN-short-slug.md` (lowercase, hyphenated slug).

## Lifecycle tags (D098) — mandatory

Every ADR **must** tag its evidence/proof status with a D098 lifecycle tag. This
is mandatory, not optional. The tag prevents the exact drift pattern D098
targets: calling a bounded claim "validated", or presenting a proposed idea as
accepted doctrine.

Tag the **evidence backing** of the decision, not just its acceptance status:

| Tag | Meaning |
|---|---|
| `[proposed]` | Put forward; not yet proven or accepted. The default for a fresh ADR awaiting review. |
| `[bounded]` | Proven within a bounded scope (fixture, synthetic, controlled, single-case) — not broadly. |
| `[smoke]` | Exercised at runtime in a smoke / quick-check capacity, but not rigorously validated. |
| `[validated]` | Rigorously validated against source, runtime, and/or golden evidence. The strongest tag — use sparingly and honestly. |
| `[deferred]` | Explicitly deferred; the decision is recorded but not currently being advanced or proven. |

Place the tag on the ADR's status line. Example: `Status: Accepted [bounded]`.

The tag and the ADR `Status` (Proposed / Accepted / Superseded / Rejected) are
**independent dimensions**:

- **Status** describes acceptance — did we commit to this?
- **Lifecycle tag** describes evidence backing — how well is it proven?

A decision can be `Accepted` (we committed to it) and `[bounded]` (we have only
bounded proof). Mislabeling a `[bounded]` decision as `[validated]` is precisely
the drift this tag exists to prevent.

## ADR template (MADR)

Copy this template for each new ADR. Fill **every** section. An ADR with
unfilled placeholders or missing alternatives is not complete.

````markdown
---
id: ADR-NNNN
title: <short, decision-oriented title>
status: Proposed | Accepted | Superseded | Rejected
lifecycle: [proposed] | [bounded] | [smoke] | [validated] | [deferred]
date: YYYY-MM-DD
superseds: ADR-MMMM | none
related: [ADR-MMMM, D0xx]
---

# ADR-NNNN: <Title>

## Status
<status plus lifecycle tag, e.g. "Accepted [bounded]">

## Context
<What is the situation? What forces are at play? What problem does this
decision solve? Reference relevant GSD decisions (D0xx) and research artifacts
as provenance, not as proof.>

## Decision
<What is the change we are deciding on? State it unambiguously.>

## Consequences
- <What becomes easier>
- <What becomes harder>
- <What we will need to revisit>

## Alternatives Considered
### Option A: <name>
<pros / cons>
### Option B: <name>
<pros / cons>
<Alternatives are required, not optional decoration. If only one option is
viable, state why explicitly rather than omitting the section.>

## References
- <Related ADRs, GSD decisions (D0xx), research artifacts, proof gates>
````

## Workflow

1. **Significance test** — does the decision meet at least one gating criterion
   (architectural / layer-contract / new port-adapter / persistent tech choice)?
   If not, do not create an ADR.
2. **Cluster check** — is this part of a coupled decision cluster? Document it
   as one ADR, not several.
3. **Existing-ADR scan** — check the index below for conflicts or supersession.
4. **Draft as `[proposed]`** — create the file with the next zero-padded number
   and a slug; fill every section of the template.
5. **Review and accept** — an ADR may be drafted by an agent, but `Accepted`
   status and lifecycle promotion to `[validated]` require explicit review. Do
   not silently promote a `[proposed]` or `[bounded]` decision to
   `[validated]`.
6. **Link governance** — if the decision has a GSD row (`D0xx`), cross-link
   both ways (ADR → `D0xx` in References; the governance workflow may point
   `D0xx` → ADR).
