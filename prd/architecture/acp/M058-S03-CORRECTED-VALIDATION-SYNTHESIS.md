# M058/S03: Corrected ACP-kit validation synthesis and M057 erratum

## Status

M058 correction complete. The M057/S04-S05 validation conclusion was too broad and is corrected here.

## Scope and authority boundary

This artifact is a corrective diagnostic synthesis. It does not rewrite historical M057 artifacts; it supersedes the over-broad interpretation for future planning. It does not make git-lex or ACP-kit the ACP source-truth store, does not approve main `.lex` state, does not approve production adoption, and does not validate R035, R037, or R038.

## What M057 got right

M057 correctly proved:

- Canonical ACP-kit init works with `git-lex init --kit rager306/git-lex-kit-acp <workspace>`.
- ACP-kit namespace correction enables 12-class discovery.
- Positive ACP Decision fixtures can pass current `git-lex validate`.
- Positive ACP fixtures can sync and query through git-lex.
- Runtime proof can be kept in disposable workspaces without main checkout residue.
- Main `.lex`, source-truth migration, production adoption, and R035/R037/R038 validation remain blocked.

## What M057 overstated

M057/S04-S05 treated object-property literal probes as evidence that negative validation enforcement failed:

```text
acp.ValidationClaim.validatesRequirement: "R035"
acp.ValidationClaim.doesNotValidateRequirement: "R037"
acp.HealthFinding.blocksClaim: "not-an-iri-reference"
```

That interpretation was too broad.

Corrected conclusion:

```text
The M057 object-property probe was not a true validation negative. The validate path normalizes object-property frontmatter values into IRIs before SHACL validation. The sync/query surface can still expose those same values as literals. Therefore the result proves a validate/query representation mismatch, not a direct failure of `sh:nodeKind sh:IRI` validation on the validation graph.
```

## M058/S01 correction: validate graph versus query graph

M058/S01 proved:

- `cmd_validate` validates per-file Turtle generated by `frontmatter_to_turtle`.
- `frontmatter_to_turtle` detects object properties from generated shapes.
- For object properties, frontmatter values are emitted as `urn:entity:{slug}` IRIs in the validation graph.

Conceptual conversion:

```yaml
acp.ValidationClaim.validatesRequirement: "R035"
```

becomes validation Turtle equivalent to:

```ttl
<urn:doc:...> acp:validatesRequirement <urn:entity:r035> .
```

not:

```ttl
<urn:doc:...> acp:validatesRequirement "R035" .
```

Therefore `sh:nodeKind sh:IRI` passes for the validation graph.

Runtime comparison confirmed:

```text
validate_code=0
commit_code=0
sync_code=0
query_code=0
```

and query still exposed literal values on `fm:` and ACP predicate surfaces.

## Corrected validation claim

Use this wording:

```text
For the current ACP kit, `git-lex validate` can accept positive ACP records, but current generated ACP shapes are too underconstrained to prove negative enforcement. Object-property string values are normalized into IRIs for validation, while datatype, enum, and required-field constraints are absent from generated shapes. Strengthening ACP validation requires ontology domains/restrictions or generator changes, followed by a new negative runtime proof.
```

Avoid this wording:

```text
`git-lex validate` is broken.
```

Also avoid this wording:

```text
ACP-kit validation enforcement is proven.
```

## Updated downstream recommendation

Proceed with constrained L2 diagnostics, but refine the M057 recommendation as follows.

Allowed:

- Use ACP-kit/git-lex in disposable workspaces for diagnostic projection, class discovery, positive sync, and query.
- Treat positive `git-lex validate` results as acceptance smoke for currently generated shapes.
- Use query surfaces as derived diagnostic views, not source truth.
- Use M058 findings to plan validation-strengthening work.

Blocked:

- Do not use `git-lex validate` as a hard ACP proof gate yet.
- Do not claim negative validation enforcement for current ACP generated shapes.
- Do not claim ACP-kit or git-lex validates R035, R037, or R038.
- Do not initialize main checkout `.lex`.
- Do not migrate ACP source truth into git-lex or `.lex`.
- Do not approve production adoption.

Required before validation-gate adoption:

- Add `rdfs:domain` or OWL restrictions so datatype/enum/required fields attach to class shapes, or change the generator to emit intended constraints.
- Regenerate ACP shapes and verify they include `sh:datatype`, `sh:in`, or `sh:minCount` where intended.
- Build true negative fixtures that survive `frontmatter_to_turtle` normalization.
- Prove `git-lex validate` exits non-zero with actionable diagnostics for those fixtures.
- Only then consider `git-lex validate` as a candidate ACP proof gate.
