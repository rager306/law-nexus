# M043 ACP Canonical Integration Owner Decision

**Date:** 2026-05-22

## Status

Accepted owner/mode decision for `M043-qzpaay / S01`.

## Decision

M043 will prove ACP canonical integration through an **opt-in architecture-build composition step**.

The selected owner is the architecture build/report layer, not the source extractor. The existing canonical extractor remains responsible for the default tracked registry projection from curated PRD/GSD/ADR/source/runtime mappings. M043 S02 may add an opt-in composition mode that reads current canonical registry JSONL plus ACP composition staging outputs and emits canonical integration proof outputs through a checked generator path.

## Selected integration mode

Selected mode: **opt-in composition**.

The S02 implementation should add a checked integration path that:

- keeps default `scripts/extract-prd-architecture-items.py --check` behavior unchanged;
- keeps default `scripts/build-architecture-graph.py --check` behavior unchanged;
- does not hand-edit tracked canonical JSONL;
- can generate/check an ACP-composed architecture registry through explicit flags or a dedicated wrapper command;
- records owner, mode, counts, diagnostics, and non-authoritative boundary status.

## Deferred alternatives

### Extend the canonical extractor directly

Deferred for M043.

Reason: direct extractor extension would make ACP governance rows part of default canonical JSONL generation. That may be correct later, but it needs stronger source-mapping and decision-promotion semantics than M043 should introduce in one step.

### Keep ACP custom-only

Deferred as the fallback if S02 evidence fails.

Reason: M042 already proved custom-only composition staging. M043 exists specifically to test a canonical integration seam beyond custom staging, while preserving rollback to custom-only if checks fail.

## Ownership boundaries

| Layer | Owner in M043 | Role |
| --- | --- | --- |
| Source evidence | PRD/GSD/ADR/source/runtime evidence plus ACP Markdown/contracts/decisions | Authoritative source for claims and non-claims. |
| Default canonical extractor | `scripts/extract-prd-architecture-items.py` | Owns default tracked canonical JSONL generation; unchanged by S01. |
| ACP composition staging | `scripts/build-acp-composition-staging.py` | Owns custom ACP composed proof outputs and diagnostics. |
| Opt-in architecture-build composition | S02 implementation | Owns explicit ACP-composed integration proof path. |
| Default verifier | `scripts/verify-architecture-graph.py` | Must remain green on default tracked canonical registry. |

## S02 implementation contract

S02 should implement a minimal opt-in path with these properties:

1. default architecture extractor/build/verifier behavior remains green and unchanged;
2. ACP-composed outputs are generated only when an explicit flag or dedicated command is used;
3. generated outputs are not manually copied from ACP derived JSONL;
4. ACP rows in composed outputs preserve source anchors to ACP source records, contracts, or decisions;
5. decision candidates remain `decision_candidate` with `authority_required: true`;
6. proof-gate records remain definitions, not proof satisfaction;
7. output diagnostics include owner, mode, input paths, output paths, counts, diagnostics, and boundary statement;
8. `--check` detects stale ACP-composed outputs;
9. tests cover enabled and disabled modes, stale output, duplicate IDs, broken endpoints, missing non-claims, missing authority requirement, and canonical path refusal.

Suggested implementation options for S02:

- add explicit optional flags to `scripts/build-architecture-graph.py` for ACP-composed inputs and report outputs; or
- add a dedicated wrapper command that invokes existing extractor/build semantics and ACP composition staging without changing defaults.

Preferred option: dedicated wrapper or opt-in build flags, whichever is smaller and keeps default verifier behavior unchanged.

## Allowed outputs for S02

S02 may create custom proof outputs under ACP derived paths, for example:

```text
prd/architecture/acp/derived/canonical-integration.items.jsonl
prd/architecture/acp/derived/canonical-integration.edges.jsonl
prd/architecture/acp/derived/canonical-integration-report.json
```

If S2 changes tracked canonical JSONL, the change must be generated by the selected owner command and must be accompanied by default extractor/check evidence. Manual edits are not allowed.

## Blocked actions

- Do not hand-edit `prd/architecture/architecture_items.jsonl`.
- Do not hand-edit `prd/architecture/architecture_edges.jsonl`.
- Do not copy ACP canonical-shaped, integrated, or composed JSONL into tracked canonical registry JSONL.
- Do not make ACP composition the default verifier path in M043.
- Do not promote ACP decision candidates to accepted architecture decisions.
- Do not treat proof-gate definitions as proof-gate satisfaction.
- Do not use custom-path success as evidence that default tracked architecture registry is current.
- Do not initialize git-lex, RDF, SHACL, SPARQL, or dashboard authority layers in M043.

## Required non-claims

M043 does not validate:

- R035;
- R037;
- R038;
- parser completeness;
- legal correctness;
- FalkorDB ingestion or runtime loading;
- graph-vector retrieval quality;
- production readiness;
- independent external review.

## S03 decision input

S03 should decide whether the S02 proof supports one of these outcomes:

1. keep ACP canonical integration as opt-in composition proof;
2. plan a future default extractor integration milestone;
3. revert to M042 custom-only composition staging if integration semantics are too weak.

Default recommendation: keep integration opt-in unless S02 proves default canonical mutation is safe and necessary.
