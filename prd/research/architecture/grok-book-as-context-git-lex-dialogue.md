### Суть методики

Это способ **заземлить ИИ-агента** (Claude, Cursor и подобные) на **авторитетный первоисточник** — конкретную фундаментальную книгу — вместо того чтобы полагаться на «среднюю температуру интернета».

Проблема, которую решает автор: Обычные промпты дают архитектуру «среднего качества» — на уровне туториалов и типовых решений. Для production-уровня (гарантии доставки, консистентность, идемпотентность, обработка сбоев) этого недостаточно.

Решение: Превращать книгу в **структурированную, индексируемую базу знаний** (Obsidian-вики), которую агент может эффективно использовать как контекст.

### Как это работает (основные шаги)

1.   **Выбор книги** Берётся авторитетный источник по теме (пример автора — _Distributed Systems: Principles and Paradigms_ Эндрю Таненбаума).
2.   **Преобразование книги в вики** С помощью инструмента **obsidian-llm-wiki** (olw) книга нарезается на чанки, из них извлекаются концепции, создаются отдельные страницы с вики-ссылками и метаданными (confidence, sources, traceability к оригинальному тексту). Получается граф концепций, а не просто суммаризация.
3.   **Индексация** Вики индексируется гибридным поиском, чтобы агент мог быстро находить релевантные понятия.
4.   **Использование агентом** Агент получает не общий промпт, а запрос, сформулированный в терминах книги. Он «гуляет» по графу концепций и строит архитектуру, опираясь на точные определения и принципы из первоисточника.

### Пример из статьи

Задача: спроектировать интеграцию 1C-Bitrix → ERP Galaktika → CRM Creatio.

*   Без методики → агент предложит CSV/FTP или простую синхронизацию.
*   С Book-as-context (по Таненбауму) → агент выдаёт event-driven архитектуру с persistent queues, AMQP-состояниями, idempotency keys, DLQ, eventual consistency, приоритетными транспортами и т.д.

### Для кого и зачем

Методика ориентирована на тех, кто использует ИИ для проектирования сложных систем (архитекторы, tech lead’ы, AI-driven команды). Цель — получать **production-grade** решения с traceability к источнику и минимальной итерацией.

### Важные ограничения (автор прямо указывает)

*   Качество сильно зависит от книги и качества нарезки/индексации.
*   Нужен слой «агент + правильная маршрутизация запросов» — иначе это просто «слепой RAG».
*   Черновики с низким confidence требуют проверки человеком.
*   Стоимость обработки книги — около $1.5 за прогон (на момент публикации).

### Простое объяснение «на пальцах»

Представь, что ты строишь **очень сложную систему** — например, большую онлайн-магазин, который связан с складом, бухгалтерией, доставкой и другими программами. Всё должно работать чётко: заказы не теряются, данные не путаются, даже если где-то случится сбой.

Обычный ИИ (типа Claude или Cursor) — умный, но он знает «всё понемногу» из интернета. Когда ты просишь его спроектировать такую систему, он часто выдаёт **среднее решение**: «сделаем так, как у всех». В итоге могут быть скрытые проблемы — сообщения пропадут, данные рассинхронизируются, при большой нагрузке всё упадёт.

### В чём смысл подхода Book-as-context

Автор берёт **лучшую авторитетную книгу** по теме (например, классику по распределённым системам Таненбаума — это как «библия» для таких задач).

Из этой книги он делает **специальную удобную карту знаний** для ИИ:

*   Книгу разбирают на маленькие понятные кусочки-идеи.
*   Эти кусочки связывают между собой (как вики-страницы с ссылками).
*   Получается быстрый поиск: ИИ может мгновенно найти нужное правило или понятие.

Теперь, когда ты даёшь ИИ задачу («сделай интеграцию магазина с ERP»), он **не просто вспоминает общие знания**. Он заглядывает в эту «карту из книги» и строит решение, опираясь на точные принципы эксперта:

*   Как правильно гарантировать, что сообщение дойдёт.
*   Как делать систему устойчивой к сбоям.
*   Какие паттерны использовать для надёжности.

### Простая аналогия

Обычный ИИ — как хороший студент, который много читал в интернете. Book-as-context — как если бы у этого студента была **личная подробная энциклопедия от лучшего профессора**, и он реально заглядывает в неё при каждом важном решении, а не просто вспоминает «что-то где-то читал».

### Итог

Это **конкретная методика** работы с ИИ при проектировании сложных систем. Суть: вместо того чтобы полагаться на «средние» знания ИИ, ты даёшь ему **высококачественную экспертную основу** из проверенной книги и учишь правильно ею пользоваться. В результате архитектура получается более надёжной, профессиональной и соответствующей лучшим практикам.

Это не замена чтению книги человеком, а способ **заземлить ИИ** на реальные глубокие знания при проектировании.

### Кратко о видео (автор — Macro Lens, май 2026)

Название: **«4 Architectural Decisions an AI Will Never Suggest»**.

Суть: Современные ИИ (Claude, Cursor и т.д.) **всегда** предлагают **среднестатистическое, популярное решение** — потому что они предсказывают «самый вероятный следующий токен» на основе всего, что видели в обучении. А среднее в коде — посредственное.

Автор приводит 4 реальных примера решений из production (финтех, платформы), которые **ни один ИИ никогда не предложит сам**:

*   Намеренно **монолит** вместо модных микросервисов (упрощение, меньше on-call).
*   Название таблиц в **прошедшем времени** (users_created, payments_settled) — выглядит «неправильно», но делает схему append-only и безопасной.
*   **Удаление** кэша Redis вместо его тюнинга (оказалось, что Postgres уже справлялся быстрее и дешевле).
*   Отказ от популярного фреймворка после измерения реальной стоимости.

### Связь с Book-as-context Добрякова

**Очень похоже по проблеме**, которую решают оба подхода:

*   **Видео** говорит: **ИИ по умолчанию даёт «среднее»** → нужны **человеческий вкус, scar tissue и сознательный отказ** от популярного.
*   **Book-as-context** говорит: **чтобы ИИ не давал среднее**, нужно **заземлить его на авторитетный первоисточник** (книгу Таненбаума и т.п.), превратив её в структурированную вики-карту знаний.

Оба подхода решают одну и ту же боль при проектировании сложных систем:

*   Обычный промпт → ИИ выдаёт «как у всех» (микросервисы, кэш, CRUD и т.д.).
*   Нужен **выход за пределы среднего**:
    *   В видео — через **человеческий опыт и вкус**.
    *   У Добрякова — через **принудительное использование глубокой экспертной книги** как фундамента.

### Простая аналогия «на пальцах»

Представь, что ИИ — это очень умный студент, который прочитал все учебники и статьи в интернете.

*   Обычный способ → он отвечает, как «средний отличник».
*   Видео Макро Ленс → «Старший инженер иногда делает то, что студентам в голову не придёт, потому что у него есть шрамы от реальных провалов».
*   Book-as-context → «Дай этому студенту не все статьи, а одну толстую библейскую книгу по предмету, разложи её по полочкам — и пусть он сначала смотрит туда, а не в свои общие знания».

**Вывод**: Видео отлично дополняет статью Добрякова. Одно показывает **проблему** (ИИ регрессирует к среднему), второе — **конкретный инструмент**, как эту проблему частично обойти (книга как контекст + вики).

Вместе они дают мощный подход: используй Book-as-context для структуры + добавляй свой человеческий «вкус» для тех решений, которые ИИ никогда не предложит.

Как внедрить Book-as-context в CI/CD

**У меня нет надёжной, проверенной информации о готовой реализации Book-as-context именно в CI/CD от автора или публичных источников (на май 2026).**

Методика Добрякова описана только для проектирования архитектуры и генерации кода/ТЗ через агента (Claude/Cursor). Прямого упоминания CI/CD-пайплайнов в статье нет. Ниже — **точная экстраполяция** на основе доступных деталей из статьи + стандартных практик (отмечаю, где inference).

### Что можно внедрить реалистично

1.   **Wiki как артефакт репозитория** (прямо из статьи)

    *   Генерируешь wiki один раз через **olw** (obsidian-llm-wiki).
    *   Кладешь папку tanenbaum/wiki/ (или любую другую книгу) в Git-репозиторий проекта.
    *   Это уже даёт traceability: каждый файл .md содержит ссылки на source-чанк книги.

2.   **Интеграция в CI/CD как knowledge layer**

    *   **На этапе code review / architecture review**: Агент (Cursor, Claude Code, GitHub Copilot Workspace, custom MCP) получает задачу + инструкцию: «При ревью архитектуры/кода используй только концепты из wiki/[book]/ как primary source. Ссылайся на конкретные страницы». → CI-шаг: claude review --context wiki/index.md.
    *   **Quality gates на базе книги**: Создаёшь скрипт/агента, который:
        *   Парсит предлагаемые изменения (дифф PR).
        *   Через hybrid-search в wiki ищет релевантные принципы (persistent communication, idempotency, eventual consistency и т.д.).
        *   Выдаёт отчёт: «Нарушение §X Таненбаума: нет DLQ для гарантии доставки».
        *   Fail пайплайна при критических несоответствиях.

    *   **Автогенерация/валидация конфигов и IaC**: Применяешь Book-as-context при генерации Terraform/Helm/K8s-манифестов: агент строит их по принципам из книги → CI проверяет соответствие.

3.   **Практическая схема внедрения (минимальная)**

    *   Шаг 1: Сгенерировать wiki (одноразово, ~$1.5).
    *   Шаг 2: Зафиксировать в monorepo или отдельном knowledge-repo.
    *   Шаг 3: В GitHub Actions / GitLab CI / Jenkins добавить job:YAML 
```
- name: Book-as-context Architecture Check
  run: |
    python check_with_wiki.py --pr-diff $DIFF --wiki-path wiki/
``` Где check_with_wiki.py вызывает LLM с RAG по wiki.
    *   Шаг 4: Добавить в pre-merge hooks или mandatory checks.

### Ограничения (строго по методике)

*   Wiki — статичный артефакт. Автообновление при новых версиях книги требует перезапуска olw.
*   Без хорошего hybrid-search (MCP) агент будет «тонуть» в большом vault.
*   Стоимость: каждый запуск агента в CI — токены. Делать только на critical paths.
*   Это **не заменяет** человеческий review, а только заземляет ИИ-часть.

**Вывод**: Book-as-context внедряется в CI/CD **как постоянный слой знаний в репозитории + агентные проверки**, а не как отдельный пайплайн. Это логичное расширение, но прямого руководства от Добрякова на момент моих данных нет.

Если у тебя есть доступ к его другим материалам или конкретному стеку (GitHub Actions / конкретная книга), уточни — смогу дать более точный план.

Поищи аналогичные по смыслу идеи - цель зафиксировать архитектурные паттерны

**Аналогичные идеи по фиксации архитектурных паттернов (на май 2026)**

Основная цель Book-as-context — превратить авторитетный источник (книгу) в **структурированную, связанную базу знаний** (Obsidian-wiki + граф), чтобы ИИ не «выдумывал среднее», а опирался на проверенные паттерны.

Вот ближайшие по смыслу подходы:

### 1. Architectural Decision Records (ADR) + AI/RAG

Это самый распространённый и близкий аналог. ADR — короткие документы, фиксирующие **контекст + решение + последствия + альтернативы** для важных архитектурных выборов.

*   **Связь с Book-as-context**: ADR можно делать **RAG-ориентированными** — вся коллекция ADR лежит в репозитории как векторная база + markdown. Агент (Claude/Cursor) при генерации кода или дизайна сначала ищет релевантные ADR и строго следует им.
*   Многие команды уже используют AI-агентов для автоматической генерации и обновления ADR (сканирование кода → шаблон → запись).
*   Преимущество: статусы (accepted / superseded / deprecated) позволяют ИИ видеть, какие паттерны устарели.
*   Примеры: GitHub org adr.github.io, интеграции с Claude Code, LangChain-агенты с памятью на ADR.[⁠Adr.github](https://adr.github.io/)

Это **не книга**, а **живая история решений проекта** — идеально дополняет или заменяет Book-as-context на уровне команды.

### 2. LLM Wiki / Karpathy’s LLM Wiki pattern (2025–2026)

Практически **прямой аналог** Book-as-context. Андрей Карпатный описал паттерн:

*   Raw sources (книги, статьи, код) →
*   LLM генерирует связанную Obsidian-wiki (концепты, страницы, граф ссылок) →
*   Агент использует эту вики как постоянный контекст.

Сейчас есть готовые инструменты: obsidian-llm-wiki, obsidian-wiki-framework и локальные версии на Ollama. Это именно то, что Добряков делал с Таненбаумом — только в более общем виде (не обязательно одна книга).[⁠Gist.github](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

### 3. Architecture as Code (AaC / Architecture-as-Code)

Архитектуру описывают **как код** (YAML/JSON/DSL + модели + fitness functions), чтобы:

*   Версионировать,
*   Автоматически валидировать,
*   Давать агентам в контекст.
*   Финансовые и enterprise-команды строят **knowledge graphs** архитектуры и кормят их в LLM.
*   Примеры: FINOS Architecture-as-Code, Structurizr, C4 models as code, fitness-function-driven architecture.
*   Цель та же: ИИ и люди работают по одному актуальному «канону», а не по среднестатистическим знаниям.[⁠GitHub](https://github.com/finos/architecture-as-code)

### 4. Другие близкие практики

*   **Enterprise Architecture Knowledge Graphs** + RAG — паттерны и reference architectures хранятся в графе, LLM генерирует решения строго по нему.
*   **Pattern Repositories** в AI-архитектуре (например, в awesome-ai-architect) — коллекции проверенных паттернов с примерами и trade-off’ами.
*   **Living Documentation + Agentic AI** — ADR становятся «живыми»: агент периодически проверяет код на соответствие зафиксированным паттернам и поднимает флаги.

### Что выбрать и как комбинировать

| Подход | Что фиксирует | Лучше всего для | Сложность внедрения |
| --- | --- | --- | --- |
| Book-as-context | Глубокие фундаментальные паттерны (из книги) | Сложных distributed/system design | Средняя |
| ADR + RAG | Конкретные решения проекта | Командной разработки, CI/CD | Низкая |
| LLM Wiki (Karpathy) | Любые источники в граф | Личного/проектного knowledge base | Средняя |
| Architecture as Code | Исполняемую модель архитектуры | Enterprise, автоматизации | Высокая |

**Рекомендация**: Начни с **ADR + LLM Wiki** — это даст быстрый win и напрямую решает фиксацию паттернов. Потом добавь книгу по Таненбауму/Клейпману как один из raw sources.

Если нужно — могу дать конкретный минимальный шаблон внедрения (GitHub Actions + Claude + Obsidian). Уточни стек или книгу/паттерны, которые хочешь зафиксировать.

Какой архитектурный шаблон, подход лучше применить при построении графа базы знаний по нормативным актам российского законодательства, где LLM только помогает искать, но не интерпретирует граф

**Рекомендуемый подход: Ontology-Driven Legal Knowledge Graph (на базе RDF/OWL + ELI-подобной онтологии) с Hybrid Retrieval (Graph + Vector), где LLM строго ограничен retrieval-ролью.**[⁠arXiv](https://arxiv.org/html/2505.00039v3)

### Почему именно этот шаблон

*   **Нормативные акты РФ** — это иерархические, ссылочные, версионные, темпоральные данные (статьи, главы, изменения, вступление в силу, отмена). Граф отлично захватывает отношения: «ссылается на», «изменяет», «отменяет», «входит в состав», «применяется к».
*   **Требование «LLM только помогает искать, не интерпретирует»** → граф должен быть **схемным, детерминированным и queryable напрямую** (SPARQL/Cypher). LLM используется только для:
    *   перевода естественного языка в структурированный запрос;
    *   embedding-based поиска entry-point’ов;
    *   но **финальный ответ формируется строго из результатов графа** без дополнительной генерации/интерпретации.

Это минимизирует риски галлюцинаций и обеспечивает traceability (каждый факт traceable до исходного текста нормативного акта).

### Основная архитектура (проверенные паттерны из legal domain)

1.   **Ontology Layer (концептуальная схема)** Используйте/расширьте стандартные legal ontologies:
    *   **ELI (European Legislation Identifier)** + **FRBRoo** (Work / Expression / Manifestation) — идеально для версий нормативов и их текстовых реализаций.
    *   Расширьте для РФ: классы NormativeAct, Article, Chapter, Version, Amendment, Jurisdiction=RF, EffectiveDate, RepealedBy и т.д.
    *   SHACL для валидации графа (чтобы структура всегда была корректной).[⁠Penni.wu](https://penni.wu.ac.at/supervision/Erwin%20Filtz%20Thesis%202021.pdf)

2.   **Graph Storage**
    *   **Neo4j** (удобен для traversal и Cypher) или **RDF triple-store** (Blazegraph, GraphDB, Apache Jena) — для SPARQL.
    *   В РФ уже есть успешный пример: Knowledge Graph по реестру юрлиц на RDF/OWL + SPARQL (миллиарды триплов).[⁠Ceur-ws](https://ceur-ws.org/Vol-1963/paper497.pdf)

3.   **Ingestion Pipeline (строительство графа)**
    *   Парсинг официальных источников (КонсультантПлюс XML, Гарант, pravo.gov.ru).
    *   LLM помогает: извлекает сущности и отношения **на этапе построения** (одноразово или при обновлении), но результат обязательно валидируется/записывается в схемный граф.
    *   Иерархическая сегментация: Document → Part → Chapter → Article → Point.
    *   Связи: citations, amendments, temporal (действует с… по…).

4.   **Retrieval Layer (где работает LLM)**
    *   **Hybrid Search**:
        *   Vector search (embeddings статей) → находит entry-points.
        *   Graph traversal от entry-points (Cypher/SPARQL) → собирает связанный контекст (все ссылки, версии, связанные акты).

    *   LLM только:
        *   Преобразует вопрос пользователя → SPARQL/Cypher или векторный запрос.
        *   Не получает сырой граф для «свободной интерпретации».

    *   Вывод: чистые данные из графа + цитаты из оригинального текста.

5.   **Дополнительные паттерны**
    *   **Hierarchical + Temporal Graph** — для версий нормативов (каждый акт имеет узлы Version).
    *   **Multi-Graph** (если нужно): один граф для структуры документов, второй — для понятий/терминов.
    *   **Living Graph** с периодическим обновлением (ETL при выходе новых редакций).

### Сравнение альтернатив

| Подход | Подходит? | Почему да/нет для вашего случая |
| --- | --- | --- |
| Чистый Vector RAG | Слабо | Теряет связи между актами |
| GraphRAG (Microsoft) | Хорошо | Но в классическом виде LLM может интерпретировать — нужно доработать под «retrieval-only» |
| Ontology-Driven KG + SPARQL | **Лучший** | Максимальная детерминированность и traceability |
| ADR / Book-as-context | Дополнение | Можно добавить как слой поверх графа (фиксированные архитектурные решения) |

### Практические шаги внедрения

1.   Выбрать базу (Neo4j или GraphDB).
2.   Создать/импортировать ontology (начните с ELI + российские расширения).
3.   Построить ETL-пайплайн (LLM только на extraction + human-in-loop validation).
4.   Реализовать hybrid retriever (LangChain/LlamaIndex + graph query layer).
5.   Обернуть в API: пользователь → LLM → structured query → graph → ответ с источниками.

Этот подход уже применяется в legal AI-проектах по всему миру и имеет российские прецеденты. Он даёт **детерминированность**, масштабируемость и полное соответствие вашему ограничению на LLM.

Если укажите объём данных, стек (Neo4j / Python / etc.) или конкретные нормативные области — дам более детальный шаблон/пример Cypher/SPARQL.

SHACL validation for legal graphs

**SHACL (Shapes Constraint Language)** — это W3C-рекомендация (2017) для **валидации RDF-графов** путём описания ограничений (shapes). Она идеально подходит для legal knowledge graphs, особенно при работе с нормативными актами РФ, где требуется строгая структура, traceability и детерминированность.[⁠W3](https://www.w3.org/TR/shacl/)

### Почему SHACL критически важен именно для legal graphs

*   Нормативные акты имеют жёсткую иерархию, версионность, темпоральные свойства и ссылки → SHACL гарантирует, что данные соответствуют онтологии (ELI + расширения для РФ).
*   LLM **не интерпретирует** граф — SHACL обеспечивает **закрытый мир** (Closed World Assumption): всё, что не указано явно, считается нарушением.
*   Поддерживается большинством трипл-сторов: GraphDB, Apache Jena, Stardog, Neo4j (с расширениями), AllegroGraph.[⁠Ontotext](https://www.ontotext.com/knowledgehub/fundamentals/what-is-shacl/)

### Основные элементы SHACL для legal graphs

1.   **NodeShape** — ограничения на узел (класс).
2.   **PropertyShape** — ограничения на свойства.
3.   **Targets** — на какие узлы применяется shape (targetClass, targetSubjectsOf и т.д.).
4.   **Constraints**:
    *   cardinality (minCount / maxCount)
    *   datatype / class / nodeKind
    *   value range (minInclusive, pattern)
    *   logical (and / or / not)
    *   SHACL-SPARQL (для сложных бизнес-правил, например, проверка темпоральной консистентности)

### Пример shapes для нормативного акта (Turtle)

turtle

```
@prefix ex: <http://example.org/legal/> .
@prefix sh: <http://www.w3.org/ns/shacl#> .
@prefix eli: <http://data.europa.eu/eli/ontology#> .

# NodeShape для нормативного акта
ex:NormativeActShape a sh:NodeShape ;
    sh:targetClass ex:NormativeAct ;
    sh:property [
        sh:path eli:identifier ;
        sh:minCount 1 ; sh:maxCount 1 ;
        sh:datatype xsd:string ;
    ] ;
    sh:property [
        sh:path eli:has_version ;
        sh:minCount 1 ;
        sh:node ex:VersionShape ;
    ] ;
    sh:property [
        sh:path eli:amends ;          # ссылка на изменяемый акт
        sh:class ex:NormativeAct ;
    ] .

# Shape для версии
ex:VersionShape a sh:NodeShape ;
    sh:property [
        sh:path eli:effective_date ;
        sh:minCount 1 ; sh:maxCount 1 ;
        sh:datatype xsd:date ;
    ] .
```

### Практика применения в legal domain

*   **ELI Validator** (Европейский стандарт) — уже использует SHACL для валидации законодательных документов.[⁠Webgate.europa](https://webgate.ec.europa.eu/eli-validator/documentation)
*   Примеры из госсектора: валидация деклараций в Бельгии (DIMONA) — преобразование XML → RDF + SHACL + сложные бизнес-правила через SPARQL.[⁠YouTube](https://www.youtube.com/watch?v=1oBxjW_Rq0E)
*   В РФ: применимо к данным из pravo.gov.ru / КонсультантПлюс — парсинг → RDF → SHACL-валидация при каждом обновлении.

### Интеграция в ваш пайплайн

1.   ETL: LLM извлекает сущности → human/review → запись в граф → **SHACL validation**.
2.   CI/CD: запуск валидации при каждом обновлении (GraphDB / pySHACL).
3.   Violation Report: SHACL выдаёт детальный отчёт (resultSeverity, sourceShape, focusNode, message) — легко логировать и исправлять.
4.   Комбинация: SHACL + OWL (онтология) + SHACL Rules (SHACL 1.2).

**Ограничения** (точные):

*   SHACL Core не покрывает всё (например, сложные лицензионные проверки в естественном языке) → используйте SHACL-SPARQL.
*   Производительность на очень больших графах требует оптимизации (Trav-SHACL, parallel validation).[⁠Vldb](https://www.vldb.org/pvldb/vol17/p3589-acosta.pdf)

SHACL — **стандарт де-факто** для обеспечения качества legal knowledge graphs. Он даёт детерминированную валидацию без доверия LLM на этапе интерпретации.

Если нужны готовые shapes под конкретную онтологию (ELI + РФ), примеры Cypher/SPARQL или скрипт на pySHACL — укажите детали.

Меня интересует больше скелет архитектуры, как удержать и проверять , а не как построить -давай вернёмся к книге

**Скелет архитектуры Book-as-context (фокус на удержании и проверке, а не на начальной сборке)**

По материалам Добрякова (статья на dobryakov.com) и связанным инструментам (olw — форк obsidian-llm-wiki) на май 2026 основной подход именно **статичный, версионируемый граф знаний в Obsidian-vault**, который живёт в Git-репозитории. LLM не перестраивает всё каждый раз, а только поддерживает.

### Основные слои (скелет)

1.   **Raw layer** (неизменяемый фундамент)
    *   Чанки исходной книги (PDF → markdown) в raw/.
    *   Никогда не трогаются вручную после первого ingest.
    *   Хранят traceability (ссылки на оригинальные страницы/разделы).

2.   **Source layer** (wiki/sources/)
    *   Одна страница на чанк книги.
    *   Содержит summary + список extracted concepts.
    *   Метаданные (YAML-frontmatter): confidence, version книги, дата прогона.

3.   **Concept layer** (wiki/.drafts/ или wiki/)
    *   Отдельные страницы на каждое понятие (persistent communication, idempotency и т.д.).
    *   Wikilinks между концептами + backlinks к source.
    *   Это основной граф, по которому ходит агент.

4.   **Index & Metadata**
    *   index.md — оглавление всех источников и high-level concepts.
    *   wiki.toml / CLAUDE.md / schema — правила поведения агента (что можно/нельзя, приоритеты).
    *   log.md — журнал всех запусков olw.

### Как удерживать (maintenance)

*   **Incremental updates** (главная фича olw и Karpathy-style): При изменении одного raw-чанка перекомпилируются только связанные concept-страницы (не весь vault). Команда: olw maintain или olw run --incremental.
*   **Self-maintenance команды** (из инструментов):
    *   olw lint — отчёт об orphans (висячие страницы), broken links, stale concepts.
    *   olw maintain --fix — автоисправление alias-ссылок, создание stub-страниц, нормализация.
    *   Агент (Claude/Cursor) по расписанию или в CI запускает lint + fix.

*   **Git + CI/CD как backbone**: Vault в репозитории → каждый PR/PR-review триггерит olw lint. Fail на критических broken links или низком confidence. Human-in-loop только для спорных изменений книги (новая редакция).

### Как проверять (validation)

*   **Traceability enforcement**: каждая concept-страница обязательно содержит ссылку [[Source_XXXX]] + confidence score. Агент обязан цитировать при ответе.
*   **SHACL-подобная валидация** (можно добавить поверх): Простые правила в schema (например, каждая concept должна иметь min 1 link к source и min 2 outgoing wikilinks). LLM-агент + скрипт проверяет compliance.
*   **Periodic health checks**:
    *   Contradictions (LLM сравнивает страницы).
    *   Orphan concepts.
    *   Устаревшие утверждения (при смене редакции книги — diff raw и перекомпиляция).

*   **Hybrid-search MCP** (из статьи Добрякова): перед использованием агент сначала делает retrieval → проверяет relevance score и source freshness.

### Ограничения (точно по источникам)

*   Полная перекомпиляция при смене редакции книги — дорого (~$1.5) и требует human review.
*   Без maintain и lint vault деградирует (broken links растут).
*   LLM никогда не интерпретирует сам — только по графу + schema.

**Вывод (скелет в одной фразе)**: Raw (неизменяемый) → Source → Concept-graph в Git + olw maintain/lint в CI + schema-правила → агент ходит строго по этому графу с traceability. Это не живое самообучающееся существо, а **замороженная экспертная база с автоматизированной гигиеной**.

Если нужно — могу набросать конкретный .toml / schema или пример GitHub Actions для maintenance. Уточни книгу или домен (нормативка РФ или distributed systems).

Вопрос был именно как построить архитектурную систему для для agentic temporal graph knowledge base по российскому законодательству для отражения архитектурных решений , чтобы агент не плыл при разработке и можно было проконтролировать

**Архитектурный скелет системы: Agentic Temporal Graph Knowledge Base по нормативным актам РФ с контролем архитектурных решений**

Я опираюсь на проверенные паттерны (ELI/FRBRoo + RDF/OWL + SHACL + temporal extensions, примеры из legal KG проектов, включая российские RDF-графы реестров). Прямого готового open-source решения именно под «agentic temporal graph + архитектурные решения РФ» у меня нет — это синтез.

### 1. Общая структура (слои)

**Layer 0 — Источники (immutable)**

*   Официальные данные: pravo.gov.ru, XML/JSON выгрузки КонсультантПлюс/Гарант, тексты законов.
*   Версионирование на уровне документов (каждая редакция — отдельный Expression по FRBRoo).

**Layer 1 — Temporal Knowledge Graph (ядро)**

*   **Онтология**: расширенная ELI + FRBRoo + российские классы (НормативныйАкт, Статья, Пункт, Изменение, Версия, ДатаВступления, ДатаПрекращения, СвязьССубъектом и т.д.).
*   **Temporal модель**:
    *   Свойства validFrom, validTo, version, amendmentOf.
    *   Каждая сущность имеет временной интервал действия.
    *   Граф хранит историю изменений (кто/когда/что изменил).

*   Хранилище: **GraphDB / Blazegraph / Neo4j** (с поддержкой temporal или отдельным слоем).

**Layer 2 — Architectural Decision Layer (отражение архитектурных решений)**

*   Каждый архитектурный выбор (паттерн, trade-off, запрет) — узел ArchitecturalDecision (аналог ADR, но в графе).
*   Связи: decidesOn (какой паттерн выбран), basedOn (какой раздел закона/книги), appliesTo (какие компоненты системы), supersedes.
*   Обязательные свойства: rationale, alternatives considered, status (proposed/accepted/deprecated), date, author, confidence.
*   Это **единственный источник истины** для агента — он не может придумывать паттерны.

**Layer 3 — Agentic Layer (с контролем)**

*   Агенты (CrewAI / LangGraph / custom) имеют **роли** и **policy guardrails**.
*   Каждый агент **обязан**:
    1.   Сделать structured query в граф (SPARQL/Cypher).
    2.   Получить только validated данные.
    3.   Сослаться на конкретные ArchitecturalDecision узлы.

*   LLM используется **только** для:
    *   Перевода NL → structured query.
    *   Суммаризации результатов (без интерпретации).

*   Запрещено: свободная генерация архитектуры без ссылки на граф.

**Layer 4 — Validation & Control Plane**

*   **SHACL shapes** (обязательно):
    *   Каждая ArchitecturalDecision должна иметь basedOn → источник.
    *   Temporal consistency (нельзя иметь overlapping valid intervals без явного supersedes).
    *   Cardinality и datatype проверки.

*   **Versioning & Audit**: весь граф в Git (или DVC-like). Каждый коммит — traceable.
*   **CI/CD gates**:
    *   shacl validate перед merge.
    *   Проверка наличия ArchitecturalDecision для новых компонентов.
    *   Логирование всех действий агента с ссылкой на узлы графа.

*   **Human-in-the-loop точки**: approval для новых ArchitecturalDecision и изменений в онтологии.

### 2. Как агент «не плывёт»

*   **Принудительная маршрутизация**: агент начинает любой reasoning с retrieval из графа + ArchitecturalDecision layer.
*   **Query guardrails**: whitelist разрешённых SPARQL/Cypher шаблонов. Всё остальное — reject.
*   **Temporal awareness**: агент всегда получает данные с учётом validFrom/validTo на текущую дату.
*   **Drift detection**: периодический job сравнивает реальное состояние системы с ArchitecturalDecision узлами и поднимает алерты/блокирует деплой.

### 3. Контроль и наблюдаемость

*   **Traceability end-to-end**: от вопроса пользователя → structured query → узлы графа → ArchitecturalDecision → сгенерированный код/конфиг.
*   **Audit log**: кто (агент/человек), когда, какой query, какие решения использованы.
*   **Dashboard**: визуализация графа (темпоральные срезы + ArchitecturalDecision overlay).
*   **Policy as Code**: правила в SHACL + отдельный слой политик (например, «все интеграции должны использовать persistent communication из Таненбаума»).

### 4. Что я рекомендую как минимальный старт

1.   Онтология на базе ELI + FRBRoo + минимальные российские расширения.
2.   Граф в GraphDB + SHACL shapes (начать с cardinality, required properties, temporal consistency).
3.   Слой ArchitecturalDecision как отдельный namespace.
4.   Агентский фреймворк с обязательным retrieval + reference к решениям.
5.   Git + CI с SHACL validation + custom checks на наличие ArchitecturalDecision.

Это даёт **детерминированную, контролируемую, версионируемую** систему, где агент работает строго внутри границ, заданных архитектурными решениями и законодательством.

Если нужно — могу дать пример структуры SHACL shapes для temporal + ArchitecturalDecision или high-level schema онтологии. Уточните приоритет (temporal consistency, ArchitecturalDecision модель, agent guardrails).

SHACL Shapes для временных ограничений

**SHACL Shapes для временных (temporal) ограничений в legal knowledge graph** (на базе ELI/FRBRoo + российские расширения).

### 1. Базовые Core-ограничения (простые, без SPARQL)

turtle

```
@prefix sh: <http://www.w3.org/ns/shacl#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix ex: <http://example.org/legal/> .
@prefix time: <http://www.w3.org/2006/time#> .   # или ваш префикс validFrom/validTo

# NodeShape для сущности с периодом действия (Версия акта, Статья и т.д.)
ex:VersionWithTemporalShape a sh:NodeShape ;
    sh:targetClass ex:Version ;   # или ex:NormativeAct, ex:Article и т.д.
    sh:property [
        sh:path ex:validFrom ;
        sh:minCount 1 ;
        sh:maxCount 1 ;
        sh:datatype xsd:dateTime ;   # или xsd:date
        sh:message "validFrom обязателен и должен быть датой/датой-временем" ;
    ] ;
    sh:property [
        sh:path ex:validTo ;
        sh:maxCount 1 ;
        sh:datatype xsd:dateTime ;
        sh:message "validTo должен быть датой/датой-временем (или отсутствовать для бессрочных)" ;
    ] ;
    sh:property [
        sh:path ex:validFrom ;
        sh:nodeKind sh:Literal ;
        sh:or ( [ sh:datatype xsd:date ] [ sh:datatype xsd:dateTime ] ) ;
    ] .
```

### 2. Проверка validFrom ≤ validTo (SHACL-SPARQL — обязательно для temporal)

turtle

```
ex:TemporalConsistencyShape a sh:NodeShape ;
    sh:targetClass ex:Version ;
    sh:sparql [
        a sh:SPARQLConstraint ;
        sh:message "validFrom должен быть раньше или равен validTo" ;
        sh:prefixes [ ... ] ;   # ваши префиксы
        sh:select """
            SELECT $this ?validFrom ?validTo WHERE {
                $this ex:validFrom ?validFrom .
                $this ex:validTo ?validTo .
                FILTER (?validFrom > ?validTo)
            }
        """ ;
    ] .
```

### 3. Запрет пересекающихся интервалов (overlapping) для одной сущности/группы

(Классическая задача для версий одного нормативного акта.)

turtle

```
ex:NoOverlappingVersionsShape a sh:NodeShape ;
    sh:targetSubjectsOf ex:versionOf ;   # или targetClass
    sh:sparql [
        a sh:SPARQLConstraint ;
        sh:message "Версии одного акта не должны иметь пересекающиеся периоды действия" ;
        sh:select """
            SELECT $this ?v1 ?v2 ?from1 ?to1 ?from2 ?to2 WHERE {
                ?act ex:hasVersion ?v1, ?v2 .
                FILTER (?v1 != ?v2)
                ?v1 ex:validFrom ?from1 ; ex:validTo ?to1 .
                ?v2 ex:validFrom ?from2 ; ex:validTo ?to2 .
                FILTER ( ?from1 < ?to2 && ?from2 < ?to1 )   # классическое пересечение
            }
        """ ;
    ] .
```

### 4. Дополнительные полезные temporal shapes

*   **validTo ≥ validFrom** уже покрыто выше.
*   **Бессрочные акты** (validTo отсутствует) — используйте sh:maxCount 1 + отдельный shape.
*   **Temporal ordering** (одна версия после другой) — через SPARQL с FILTER.
*   **Связь с Time Ontology** (time:hasBeginning / time:hasEnd) — см. готовый набор из репозитория liviorobaldo/TimeOntologyInSHACL (рекомендую импортировать как основу).

### Рекомендации по внедрению в вашу архитектуру

*   **Порядок валидации**: сначала SHACL Rules (inference) → потом Core + SPARQL shapes.
*   **CI/CD**: pySHACL или GraphDB validation перед merge ArchitecturalDecision.
*   **Human-in-loop**: violations с severity sh:Violation блокируют изменения в ArchitecturalDecision layer.
*   **Performance**: ограничивайте targets (targetClass / targetSubjectsOf), а не применяйте ко всему графу.

Эти shapes обеспечивают **строгую детерминированность** временных данных. Агент при retrieval всегда получает только валидные интервалы, и drift по времени (устаревшие решения) легко детектится.

Если нужны shapes под конкретную онтологию (ваши префиксы, ELI + validFrom/validTo, ArchitecturalDecision temporal) или полный .ttl-файл — укажите детали, и я дам готовый набор.

Так как это небольшой граф -можно ли реализовать хранение архитектуры и архитектурный скелет в ladybugdb?

**Да, LadybugDB (Ladybug) подходит для хранения архитектуры и архитектурного скелета в вашем случае.**

Поскольку граф небольшой, embedded-характер LadybugDB (форк/преемник Kuzu, property graph + Cypher, columnar storage, single .lbug-файл) даёт явные преимущества.

### Почему LadybugDB подходит

*   **Embedded + serverless**: Один файл БД, встраивается прямо в приложение/агента. Нет сервера, Docker, сети — идеально для agentic-системы, где агент должен быстро и надёжно читать граф без внешних зависимостей.
*   **Property Graph + Cypher**: Полноценная поддержка узлов, отношений, свойств и мощных запросов. Можно явно смоделировать ArchitecturalDecision, Version, validFrom/validTo, supersedes и т.д.
*   **Schema на уровне таблиц** (как в Kuzu): Node и Relationship таблицы с жёсткой структурой. Это уже даёт базовый контроль «скелета» на этапе записи (нельзя вставить отношение, которое не объявлено).
*   **Позиционирование**: Заявлено как решение для **highly regulated industries** и **agentic solutions** — именно ваш случай (российское законодательство + контроль агента).
*   **Для небольшого графа**: Минимальные overhead'ы, отличная скорость аналитических запросов, простой бэкап/версионирование (файл можно держать в Git или артефактах CI).

### Как реализовать архитектурный скелет

1.   **Модель данных (пример)**:
    *   Узлы: ArchitecturalDecision, NormativeVersion, SystemComponent, TemporalContext.
    *   Отношения: DECIDES_ON, APPLIES_TO, SUPERSEDES, BASED_ON_NORM (с validFrom/validTo на узлах или отношениях).
    *   Свойства: status (proposed/accepted/deprecated), rationale, confidence, effective_date и т.д.

2.   **Контроль и валидация** (поскольку нативного SHACL нет):
    *   На уровне схемы таблиц — обязательные свойства и типы связей.
    *   В приложении/агенте — перед записью проверять наличие ссылки на ArchitecturalDecision и непротиворечивость temporal-интервалов (через Cypher или Python).
    *   ACID-транзакции (serializable) помогают держать consistency.
    *   Для сложных temporal-правил (non-overlapping версий и т.п.) — Cypher + app-level guards или лёгкий слой проверок перед коммитом.

3.   **Хранение и контроль**:
    *   Один .lbug-файл → легко версионировать, распространять, встраивать в CI/CD.
    *   Агент работает embedded: открывает БД → делает Cypher-запросы → получает только validated-данные + ссылки на архитектурные решения.
    *   Это напрямую решает «агент не плывёт» — все архитектурные решения явно в графе, агент обязан на них ссылаться.

### Ограничения (честно)

*   Нет встроенного SHACL (это property-graph БД, а не RDF-ориентированная как GraphDB). Строгую валидацию формата придётся делать на уровне приложения или через кастомные Cypher-проверки.
*   Temporal-модель реализуется через свойства и запросы (есть примеры semantic spacetime / temporal awareness на базе Kuzu/Ladybug) — не native time-travel, но для ваших задач достаточно.
*   Для очень сложных inference-правил может потребоваться дополнительный слой (но для небольшого графа это не критично).

**Вывод**: Для небольшого графа и agentic-сценария в regulated domain LadybugDB — один из лучших вариантов среди embedded-решений. Он позволяет хранить и архитектурный скелет, и temporal-данные, и решения в одном контролируемом месте, с минимальной инфраструктурой.

Расскажи какие системы графовых баз поддерживают SHACL?

**Основные графовые базы данных (RDF triple stores), которые поддерживают SHACL (на май 2026)**

SHACL — стандарт W3C для валидации RDF-графов. Полноценная нативная поддержка (встроенная валидация, SHACL Core + часто SPARQL-констрейнты) есть в следующих системах:

### Полноценная встроенная поддержка

*   **GraphDB (Ontotext)** — одна из лучших реализаций. Поддерживает SHACL через RDF4J ShaclSail. Можно включать при создании репозитория, есть reserved graph для shapes, хорошая производительность и отчёты о нарушениях.
*   **Stardog** — отличная поддержка SHACL + инструменты в Studio (подсветка синтаксиса, диагностика, валидация). Используется для data quality в enterprise.
*   **AllegroGraph** — поддерживает SHACL с версии 6.6 (2019), с инструментами agtool shacl-validate. Хорошая интеграция с Prolog и geospatial.
*   **Apache Jena** (включая Fuseki и TDB) — поддерживает SHACL (через встроенный валидатор). Работает в embedded-режиме и как сервер.
*   **RDF4J** — базовая реализация ShaclSail, на которой построены GraphDB и другие.
*   **Oracle RDF Graph** — поддержка SHACL через Eclipse RDF4J Adapter (обновлено в 2025).

### Другие варианты

*   **Amazon Neptune** — поддерживает SHACL через RDF-режим + интеграции (не всегда нативно, часто через RDF4J).
*   **Neo4j** — **не нативно**, но через плагин **Neosemantics (n10s)** можно валидировать property graph против SHACL-шейпов (частично покрывает SHACL).
*   **Blazegraph** — ограниченная/устаревшая поддержка (в старых бенчмарках упоминалась, но сейчас не активно развивается).
*   **Virtuoso** — частичная поддержка через расширения, но не полноценная как в GraphDB/Stardog.

### Standalone-инструменты (не базы, но полезны)

*   **pySHACL** (на базе RDFLib) — чистый Python-валидатор, удобно для скриптов и CI/CD.
*   **SHACL Playground** и **TopBraid SHACLer** — для тестирования.

### Важные замечания по вашему случаю (небольшой temporal legal graph + LadybugDB)

*   **LadybugDB** (форк Kuzu) — **не поддерживает SHACL нативно**. Это property-graph БД (Cypher), а не RDF-oriented. Валидацию temporal-ограничений и ArchitecturalDecision придётся делать на уровне приложения (Cypher-проверки + Python-скрипты перед записью) или через pySHACL в ETL/CI.
*   Для **небольшого графа** с жёстким контролем архитектуры лучше всего подойдут **GraphDB** или **embedded Apache Jena** — они дают настоящую SHACL-валидацию «из коробки» + temporal-поддержку через свойства.

**Рекомендация**: Если строго нужна SHACL — выбирайте **GraphDB** (для сервера) или **Jena** (embedded). Если хотите embedded property-graph без RDF-overhead — LadybugDB + app-level validation (Cypher + pySHACL на стороне).

Смоделируй насколько будет устойчива и жесткая система если SHACL будет не на уровне база , а на уровне приложения?

**Система с SHACL на уровне приложения (а не в БД) — умеренно устойчивая и относительно жёсткая, но с заметными рисками.**

Я моделирую строго на основе известных практик (сравнение DB-constraints vs app-level validation, специфика RDF/SHACL). Прямых production-кейсов именно для небольшого temporal legal graph с ArchitecturalDecision layer у меня нет.

### Уровень жёсткости (rigidity)

*   **DB-level (GraphDB, Jena, Stardog)**: **высокая жёсткость** (9–10/10). Валидация происходит атомарно при записи (в транзакции). Невалидные данные **не попадают** в граф. Temporal-консистентность, cardinality, ссылки на ArchitecturalDecision гарантированы самой БД.
*   **App-level (pySHACL / RDFLib + ваш код)**: **средняя жёсткость** (5–7/10). Валидация — это **явный шаг в коде** перед записью/обновлением.
    *   Если все пути записи данных (агент, ETL, ручные правки, API) **обязательно** проходят через один валидатор — жёсткость приближается к 7–8/10.
    *   Если есть обходные пути (прямой insert, другой скрипт, ошибка в коде) — жёсткость падает до 4–5/10.

### Уровень устойчивости (stability / resilience к дрейфу)

*   **Плюсы app-level** (для небольшого графа):
    *   Легко реализовать (одна функция validate_and_commit(graph, shapes)).
    *   Можно добавить объяснения нарушений (explainable SHACL), логи, human-in-loop.
    *   Для temporal-правил (overlapping, validFrom ≤ validTo) — Cypher/SPARQL + pySHACL работает надёжно.
    *   В agentic-системе с единственным entry-point (ваш агент) — drift минимален при строгом код-ревью.

*   **Минусы и риски** (главные слабости):
    *   **Bypassable**: любой будущий разработчик/агент/скрипт может записать данные минуя валидацию.
    *   **Дублирование логики**: если появится второй клиент/сервис — правила нужно копировать (nightmare maintenance, как в классических DB vs app constraints).
    *   **Race conditions и partial updates**: без транзакций БД легко получить промежуточное невалидное состояние.
    *   **Performance & consistency**: валидация на большом графе в приложении медленнее и может расходиться с реальным состоянием БД.
    *   **Temporal drift**: без DB-enforcement со временем могут появиться overlapping версии или ArchitecturalDecision без basedOn.

**Модель устойчивости на 1–2 года** (при небольшом графе):

*   При **одной команде + строгий CI + единственный агент** → 70–80% от DB-level (приемлемо).
*   При росте (новые интеграции, несколько агентов) → быстро падает до 40–50% (появляются «дыры»).

### Рекомендуемая архитектура для вашего случая (LadybugDB или property-graph)

1.   **Центральный сервис-вход** (один модуль): KnowledgeBaseService.write(data) → pySHACL → temporal checks → commit.
2.   **Guardrails**: все агенты вызывают только этот сервис (никаких прямых соединений к БД).
3.   **CI/CD + periodic audit**: nightly full validation всего графа + алерты.
4.   **Логи нарушений** как RDF (validationReport) — хранить в отдельном графе.
5.   **Fallback**: если нарушение критическое — откат + уведомление.

**Вывод (точный)**: Для **небольшого контролируемого agentic-графа** по российскому законодательству app-level SHACL даёт достаточную жёсткость и устойчивость **при дисциплине**. Но это **не эквивалент** DB-level. Со временем вероятность дрейфа выше (особенно temporal и ArchitecturalDecision).

Если граф остаётся маленьким и вы готовы к строгому single-entry-point — можно жить. При любом росте — переходите на GraphDB/Jena с нативным SHACL.

Есть ли решения , которые хранят SHACL в git?

**Да, такие решения существуют и это стандартная практика.**

SHACL-шейпы — это обычные RDF-файлы (Turtle .ttl, RDF/XML, JSON-LD и т.д.), поэтому их **естественно и удобно хранить в Git** как обычный код. Ниже реальные примеры и подходы (на май 2026).

### Публичные репозитории с SHACL в Git

*   **[https://github.com/MaastrichtU-IDS/shacl-shapes](https://github.com/MaastrichtU-IDS/shacl-shapes)** — официальный репозиторий Института Data Science (Маастрихт). Shapes хранятся в .ttl-файлах, добавляются через PR, CI валидирует их.[⁠GitHub](https://github.com/MaastrichtU-IDS/shacl-shapes)
*   **[https://github.com/w3c/shacl-resources](https://github.com/w3c/shacl-resources)** — W3C-репозиторий с официальными и community SHACL-шейпами.
*   **[https://github.com/SDM-TIB/SHACL-ACL](https://github.com/SDM-TIB/SHACL-ACL)** — SHACL для контроля доступа, всё в Git.
*   Множество проектов: owl2shacl, pySHACL examples, corporate knowledge graphs и т.д.

### Как это обычно реализуется (скелет архитектуры)

1.   **Структура репозитория**text 
```
shapes/
├── core/                  # базовые онтологии (ELI, FRBRoo, temporal)
├── legal-rf/              # ваши российские нормативные shapes
├── architectural/         # ArchitecturalDecision shapes
├── temporal/              # validFrom/validTo + overlapping checks
├── validation-reports/    # примеры отчётов
└── schema.ttl             # главный shapes-graph (импортирует остальные)
``` 
2.   **CI/CD + GitOps**
    *   При каждом push/PR запускается pySHACL или Jena shacl CLI — валидация shapes самих на себе (SHACL можно валидировать SHACL).
    *   Проверка на broken links, устаревшие IRI, конфликты.
    *   Авто-генерация документации (SHACL → HTML).

3.   **Преимущества хранения в Git**
    *   Полная история изменений шейпов (кто, когда, зачем изменил).
    *   Code review через PR (критично для legal domain).
    *   Branching: dev-shapes → main → production.
    *   Версионирование: теги v1.2-legal-2026-05.
    *   Легко распространять: один git clone — вся онтология + шейпы у агента.
    *   Интеграция с Book-as-context / LLM Wiki: shapes лежат рядом с Concept-graph.

### Инструменты, которые это поддерживают

*   **pySHACL** — загружает shapes прямо из папки Git.
*   **Apache Jena** / **GraphDB** — можно загрузить shapes-graph из Git-артефакта или напрямую из репозитория.
*   **TopBraid SHACL API**, **oxirs-shacl** и другие.
*   GitOps-флоу: ArgoCD / Flux подтягивают shapes в БД при мердже.

**Вывод для вашего случая (небольшой temporal legal graph + ArchitecturalDecision)**: Хранить SHACL-шейпы в Git — **лучшее и самое жёсткое решение** при app-level валидации. Это даёт traceability, audit и контроль дрейфа без нативного SHACL в БД (LadybugDB или другая).

Агент всегда работает с актуальной версией shapes из Git → валидация → commit. Drift практически исключён при нормальном CI.

Кристаллизуй идею построения архитектуры, хранения, валидация, изменения архитектуры применяя онтологии, SHACL и прочее

**Кристаллизованная идея: Governed Temporal Architectural Knowledge Graph (GTAKG)**

Это управляемая, темпоральная, архитектурно-ориентированная система знаний для российского законодательства (и связанных архитектурных решений), где **онтология + SHACL + Architectural Decisions** выступают как единый слой жёсткого контроля.

### 1. Философия (Core Principles)

*   **Single Source of Truth с версионированием**: Онтология, SHACL-шейпы и Architectural Decisions хранятся в Git (как код). Граф в БД — это лишь исполняемая проекция.
*   **Governed Evolution**: Архитектура меняется **только** через явно задокументированные и валидированные Architectural Decisions. Никакого «просто поменяли».
*   **Temporal Integrity by Default**: Всё имеет validFrom/validTo. Пересечения, ретроактивные изменения и «плавающие» решения запрещены.
*   **Agent Guardrails**: Агент **не может** свободно интерпретировать или предлагать архитектуру. Он обязан ссылаться на существующие решения и проходить валидацию.
*   **Defence in Depth**: Валидация на нескольких уровнях (Git CI → App pipeline → БД, если поддерживает).

### 2. Слои архитектуры

**Layer 0 — Источники (Immutable)**

*   Официальные тексты НПА (pravo.gov.ru и выгрузки).
*   Базовые онтологии (ELI + FRBRoo + российские расширения).

**Layer 1 — Онтология + SHACL (в Git)**

*   Расширенная онтология: NormativeAct, Version, Article, ArchitecturalDecision, TemporalContext, DecisionRationale и т.д.
*   SHACL-шейпы (в Git-репозитории):
    *   Структурные (cardinality, datatypes, обязательные ссылки).
    *   Temporal (validFrom ≤ validTo, non-overlapping версий одного акта, consistency интервалов).
    *   Архитектурные (каждое ArchitecturalDecision должно иметь basedOn, rationale, status).

*   Всё версионируется, ревьюится через PR, тестируется в CI (pySHACL + custom temporal checks).

**Layer 2 — Граф (Runtime)**

*   **Рекомендация**: LadybugDB (embedded, быстрый, для agentic) **или** GraphDB (если критично нативный SHACL).
*   Модель: Property Graph или RDF (через интеграцию).
*   Temporal: свойства validFrom/validTo + version + supersedes.
*   Architectural Decisions как полноценные узлы графа, связанные со всеми затронутыми сущностями.

**Layer 3 — Governance & Validation Pipeline**

*   **Git** → shapes + ontology + Architectural Decisions.
*   **CI** → валидация SHACL + temporal consistency + ссылки на решения.
*   **App Pipeline** (single-writer сервис):
    *   Принять proposed change.
    *   Создать/обновить ArchitecturalDecision.
    *   Проверить SHACL (pySHACL).
    *   Применить к графу с temporal marking.
    *   Только после этого — commit.

*   **Agent Layer**: Агент делает **только** controlled queries. Любой ответ должен содержать ссылки на используемые ArchitecturalDecision.

### 3. Как происходит изменение архитектуры (Governed Change)

1.   **Инициация**: Появляется потребность (новый закон, проблема в интеграции, требование compliance).
2.   **Создание Architectural Decision** (в Git):
    *   Context, Decision, Rationale, Alternatives considered, Consequences, basedOn (ссылки на онтологию/SHACL/НПА).
    *   Статус: proposed.

3.   **Review + Merge** в main.
4.   **Pipeline**:
    *   Валидация SHACL.
    *   Temporal consistency check.
    *   Применение к графу (создание/обновление связей + temporal метки).

5.   **Deployment**: Агент видит обновлённое состояние только после успешной валидации.

### 4. Уровни жёсткости и устойчивости

*   **Максимальная жёсткость**: GraphDB + нативный SHACL + single-writer pipeline.
*   **Хорошая жёсткость для agentic** (рекомендую): LadybugDB + строгий App Pipeline + SHACL в Git + CI + pySHACL перед записью.
*   App-level SHACL даёт приемлемую устойчивость при **одном контролируемом entry point**. При росте — риск дрейфа выше, чем при DB-level.

### 5. Ключевые артефакты

*   Git-репозиторий legal-architecture-knowledge (shapes, ontology, decisions, docs).
*   БД: LadybugDB (embedded) или GraphDB.
*   Validation Report как RDF (хранится и анализируется).
*   Архитектурный Decision как первый класс сущности графа.

**Итоговая кристаллизация**: Это не просто «граф + SHACL». **ArchiMate — открытый стандартный язык моделирования архитектуры предприятия (Enterprise Architecture).**

Это независимый графический язык от The Open Group (тот же консорциум, что выпускает TOGAF). Он позволяет однозначно описывать, анализировать и визуализировать связи между бизнес-доменами, приложениями, технологиями и мотивацией.[⁠Opengroup](https://www.opengroup.org/archimate-forum/archimate-overview)

### Основная структура (ArchiMate Framework)

Язык делит архитектуру на **слои** и **аспекты** (матрица 3×3 + расширения).

**Core Layers**:

*   **Business Layer** — бизнес-процессы, услуги, роли, организационная структура, ценностные потоки.
*   **Application Layer** — приложения, компоненты, интерфейсы, сервисы.
*   **Technology Layer** — инфраструктура, узлы, устройства, сети.

**Дополнительные слои** (Full Framework):

*   **Strategy Layer** — стратегия, курсы действий, capabilities.
*   **Physical Layer** — физическое оборудование, материалы, распределение.
*   **Implementation & Migration** — проекты, программы, плато (plateaus), gaps.
*   **Motivation** — цели, драйверы, требования, принципы, stakeholders.

**Аспекты** (горизонтальные):

*   **Active Structure** — кто/что выполняет (Actor, Role, Component).
*   **Behaviour** — что происходит (Process, Function, Service).
*   **Passive Structure** — на чём/с чем (Object, Data, Artifact).

Это даёт 9+ ячеек для моделирования.[⁠Wikipedia](https://en.wikipedia.org/wiki/ArchiMate)

### Ключевые элементы и отношения

*   **Элементы** — стандартизированные иконки (Business Actor, Application Service, Node и т.д.).
*   **Отношения** — 10+ типов: Realization (реализация), Serving (обслуживание), Triggering (триггер), Association, Composition, Aggregation, Influence и др.
*   **Viewpoints** — готовые перспективы (например, Application Usage Viewpoint, Technology Infrastructure Viewpoint) для разных стейкхолдеров.

### Как это работает в практике

ArchiMate идеально дополняет TOGAF: TOGAF даёт процесс и содержание, ArchiMate — нотацию и визуализацию. Используется для:

*   Анализа влияния изменений (impact analysis).
*   Выравнивания бизнеса и ИТ.
*   Документирования архитектурных решений (ADR + ArchiMate).
*   Коммуникации с руководством (одна модель — разные viewpoints).

**Инструменты**:

*   Бесплатный: Archi (open source).

### 1. Основные доступные решения (проверенные)

В чем назначение squad

**Назначение squad (git-lex-kit-squad)**

**squad** — это **kit (набор онтологии и шаблонов)** для **git-lex**, специально предназначенный для **многоагентного сотрудничества** (multi-agent collaboration) между людьми (**carbon agents**) и ИИ-агентами (**silicon agents**).

### Основная цель

Создать **общую разделяемую среду** (shared substrate), где:

*   Человеческие и ИИ-агенты совместно планируют, принимают решения, общаются и отслеживают работу.
*   Всё хранится в Git как Markdown-файлы с типизированным frontmatter.
*   Граф (RDF + SHACL) обеспечивает структуру, валидацию и SPARQL-запросы.

**Squad-репозиторий** — это **общий** (shared), а не личный. У каждого агента есть отдельный **soul-репозиторий** (личная память, журнал и т.д.).

### Основные типы документов (Document Types)

**Координация:**

*   **Agent** — идентификация участника (человек или ИИ).
*   **Pod** — временная рабочая группа внутри squad.
*   **Message** — persistent-коммуникация между агентами (запрашиваемая).
*   **Proclamation** — официальное постановление «Короля Squad».

**Работа:**

*   **Project** — что строит squad.
*   **Task** — задачи со статусом и assignee.
*   **Decision** — ключевой класс: решения с контекстом, rationale, alternatives и outcome (аналог вашего **ArchitecturalDecision**).

**Знания:**

*   **Brief** — proactive-исследования, обзоры, конкурентный анализ.
*   **Discovery** — случайные находки.
*   **Situation** — статус-отчёты (фокус, blockers и т.д.).

**Catch-all:**

*   **Freeform** — для всего, что не подходит под другие типы.

### Что **не** входит в squad

*   Личная память и журнал — в **soul-kit**.
*   Формальные исследования (Hypothesis, Experiment) — в **lab-kit**.

### Как это работает в вашей архитектуре

squad:Decision — почти готовый шаблон для **ArchitecturalDecision**. Вы можете создать свой legal-arch-kit, расширив squad:

*   Добавить temporal-свойства.
*   Импортировать ArchiMate.
*   Добавить классы для нормативных актов РФ.

**Вывод**: squad — это **операционная система для команд из людей + ИИ-агентов**, где Git становится governed knowledge base с жёсткой структурой (OWL → auto-SHACL). Именно то, что нужно для вашего Governed Temporal Architectural Knowledge Graph.

Как интегрировать lex-git с pi/GSD ?

**Интеграция git-lex с pi/GSD (Get Shit Done + Pi SDK)**

На май 2026 **прямой готовой интеграции** между git-lex и GSD/pi не существует (ни в официальных репозиториях, ни в документации). Однако оба инструмента идеально дополняют друг друга, и интеграция реализуется **естественно и мощно**.

### Что такое pi/GSD

*   **GSD** (Get Shit Done) — мета-промптинговая/агентная система для долгоживущих coding-агентов (Claude Code, OpenCode и др.).
*   **Pi SDK** — низкоуровневый harness (TypeScript), на котором построен GSD v2/v3. Даёт прямой контроль над контекстом, git-ветками, memory, dispatch, recovery и т.д.
*   GSD работает в одном Git-репозитории проекта, создаёт .gsd/, memories, decisions, knowledge graph (внутренний) и чистую историю коммитов.

### Как интегрировать (рекомендуемый подход)

**Вариант 1. Git-lex как governed knowledge layer внутри GSD-проекта (рекомендую)**

1.   В корне GSD-проекта:Bash 
`git lex init --kit legal-arch     # ваш кастомный kit (на базе squad + ArchiMate)` 
2.   GSD-агент получает доступ к git-lex:
    *   Через shell-команды (git lex query "SPARQL...", git lex create ArchitecturalDecision ..., git lex save).
    *   GSD уже умеет работать с git и shell — добавьте в его skills/harness вызовы git-lex.

3.   **Bidirectional flow**:
    *   **GSD → git-lex**: Каждый Architectural Decision / Milestone / Verdict, созданный GSD, автоматически материализуется как rf:ArchitecturalDecision (frontmatter + Markdown). GSD-агент вызывает git lex save.
    *   **git-lex → GSD**: При старте milestone/agent-session GSD делает git lex query и инжектирует в контекст:
        *   Актуальные Architectural Decisions (с temporal-фильтром).
        *   Нормативные акты РФ.
        *   ArchiMate-элементы.

    *   Knowledge Graph GSD (memories, decisions) проецируется в RDF git-lex для SHACL-валидации.

**Вариант 2. GSD как execution engine, git-lex как single source of truth**

*   В GSD-конфиге (.gsd/config или preferences) добавить post-milestone hook:TypeScript 
```
// pseudo
onMilestoneComplete: async () => {
  execSync('git lex save "GSD milestone complete"');
  execSync('git lex query "... extract decisions ..."');
}
``` 
*   git-lex-kit хранит онтологию, SHACL, ArchiMate и российские нормативные классы.
*   GSD использует git-lex как **external memory / governance layer** (чтобы агент не плыл).

**Вариант 3. Hybrid kit (самый мощный)** Создать legal-arch-kit с:

*   Импортом squad (для Decision/Task/Project).
*   Импортом ArchiMate ontology.
*   Temporal + ELI/FRBRoo для РФ-нормативки.
*   Harness-адаптером под Pi SDK (чтобы GSD-агент понимал dot-notation frontmatter).

### Преимущества такой интеграции

*   GSD даёт мощный autonomous coding loop + git management.
*   git-lex даёт **жёсткую governance** (SHACL, OWL, SPARQL, Git history, PR-review изменений архитектуры).
*   Architectural Decisions становятся первыми классами, валидируются автоматически.
*   Temporal consistency + ArchiMate + Book-as-context — всё в одном репозитории.
*   Агент (GSD) **не может** нарушить архитектуру — git-lex guardrails + SHACL.

**Практические шаги для старта**

1.   Клонируйте git-lex + git-lex-kit-squad.
2.   Создайте legal-arch-kit (я могу дать шаблон legal-arch.ttl).
3.   В GSD-проекте git lex init.
4.   Добавьте в GSD skills/shell wrappers для git lex *.
5.   Настройте CI: git lex sync && pySHACL + GSD validation.

Это даёт **полностью governed agentic систему**: GSD строит код, git-lex держит архитектуру и законодательство под контролем.
