# M022 Independent Proof Review

This report records an independent read-only proof review after M022 completion. The review was requested because passing tests can be misleading when tests are empty, over-mocked, hardcoded, or only verify artifacts generated by the same implementation path.

Review method:

- focus on proof strength, vacuous tests, mocks, hardcoded assumptions, and bare artifact evidence;
- review of what is proven without trusting implementation prose.

## Summary verdict

M022 preserves its main non-claims well. It does not validate R035/R037 and repeatedly states that product retrieval quality, parser completeness, legal-answer correctness, graph-vector/HNSW, hybrid retrieval, production FalkorDB readiness, and pilot readiness remain unproven.

However, M022's current metrics proof is materially self-confirming. The ten-case representative fixture proves structural coverage, safe IDs, redaction, source-file hash anchoring, and internally encoded expected outcomes. The metrics evaluator computes all-1.0 values from fixture-provided ranks, labels, expected IDs, rejected IDs, and diagnostic codes. The local runtime check confirms model/runtime availability and vector dimension, but does not prove per-case retrieval/ranking over the M022 corpus.

## High finding — metrics are fixture-derived, not observed retrieval-derived

Severity: high.

Evidence:

- `scripts/verify-representative-evidence-span-retrieval-metrics.py` computes MRR/recall/rejection rates from fixture fields such as `rank`, `expected_label`, `expected_candidate_ids`, `expected_rejected_candidate_ids`, and `expected_diagnostic_codes`.
- The runtime boundary check confirms `deepvk/USER-bge-m3`, status, vector dimension, and safety flags, but does not score or rank the M022 candidate set.
- The proof JSON therefore demonstrates perfect fixture-encoded metrics, not observed retrieval behavior.

Impact:

A reader could mistakenly treat M022 as evidence that `deepvk/USER-bge-m3` retrieved or ranked the representative corpus correctly. The more precise claim is: fixture-derived deterministic metrics passed, and the approved local runtime boundary was separately confirmed.

Recommended remediation:

- Add an observed retrieval-output proof in a future milestone.
- Emit observed ranked candidate IDs and diagnostics from actual scoring/retrieval.
- Compute metrics by comparing observed outputs to expected outputs.
- Keep injected runtime JSON for unit tests only, not acceptance proof.

## Medium finding — source anchors do not prove candidate provenance

Severity: medium.

Evidence:

- The corpus verifier validates declared source artifact paths and SHA-256 hashes.
- Candidate checks only require that `source_artifact` is declared and that `source_case_id` has a `CASE-` prefix.
- The verifier does not parse referenced source artifacts to confirm that `source_case_id`, `source_record_ids`, `evidence_span_id`, `source_block_id`, `citation_key`, or `act_edition_id` actually exist in those artifacts.

Impact:

A fixture can cite stable hashed files while candidate provenance is fabricated or drifted.

Recommended remediation:

- Extend the corpus verifier to load declared source artifacts.
- Validate every `source_case_id`, `source_record_id`, and citation-bearing ID against the source artifact.
- Add negative tests where source files exist and hashes match, but source IDs are bogus.

## Medium finding — final reports should separate runtime confirmation from metric computation

Severity: medium.

Evidence:

- M022 reports say metrics were evaluated using the local/open-weight runtime boundary.
- The implementation confirms runtime boundary separately, while metrics are derived from fixture fields.

Impact:

The wording is bounded by non-claims, but can still imply that the model participated in per-case ranking.

Recommended remediation:

Use wording such as:

```text
Fixture conformance and fixture-derived metric calculation passed.
The approved local runtime boundary was separately confirmed.
No per-case model retrieval/ranking over the representative corpus was proven.
```

## Medium finding — query semantics are not independently auditable

Severity: medium.

Evidence:

- The fixture stores `query_text_sha256`, not raw query text.
- The verifier only checks that each query hash is a 64-character string.
- No tracked query-hash registry or redacted query-template manifest exists.

Impact:

Bare artifacts cannot prove that the ten cases represent meaningful retrieval questions. They prove only that syntactic hash fields exist.

Recommended remediation:

Add a safe query provenance registry with:

- `query_id`;
- `query_kind`;
- redacted template ID or controlled vocabulary fields;
- normalized hash algorithm;
- expected source fixture relation;
- hash verification command.

## Low finding — tests reinforce the self-confirming proof model

Severity: low.

Evidence:

- Tests validate checked-in shape, all-1.0 expected metrics, and several fail-closed cases.
- Unit tests appropriately use injected runtime JSON for speed, but do not require observed retrieval outputs.
- Negative tests do not cover bogus source-case provenance or unregistered query hashes.

Impact:

The tests are not empty, but they mostly lock down the current proof model rather than challenging whether retrieval behavior was observed.

Recommended remediation:

Add tests that fail when:

- source IDs do not exist in declared source artifacts;
- query hashes are syntactically valid but unregistered;
- expected ranks are perfect but observed retrieval output is missing;
- runtime is confirmed but no per-case observed ranking artifact exists.

## What M022 actually proves after review

M022 proves:

- a ten-case representative fixture exists;
- required M022 classes are present;
- safe ID-shaped fields, redaction flags, non-authoritative status, and non-claims are present;
- declared source artifact paths exist and hashes match;
- verifier/test chain catches selected fixture integrity problems;
- local `deepvk/USER-bge-m3` runtime availability/dimension can be confirmed;
- fixture-derived deterministic metrics are all 1.0.

M022 does not prove:

- observed per-case retrieval/ranking by `deepvk/USER-bge-m3`;
- retrieval quality over real query semantics;
- candidate provenance against source artifacts beyond path/hash anchoring;
- product retrieval quality;
- parser-to-EvidenceSpan materialization;
- legal-answer correctness;
- graph-vector/HNSW or hybrid retrieval behavior;
- production FalkorDB readiness;
- R035 validation.

## Follow-up gates

Recommended next proof gates:

1. Observed retrieval-output proof: query source or safe query registry → local scoring/ranking → observed ranked candidate IDs → metric comparison.
2. Source provenance verifier: parse source artifacts and validate `source_case_id`, `source_record_ids`, citation IDs, and act edition IDs.
3. Query-hash provenance registry: make hashed query semantics auditable without storing raw query text.
4. Report wording cleanup: amend M022 wording or future summaries to distinguish fixture-derived metrics from runtime-observed retrieval.
5. R038 process gate: require independent proof-review before future proof milestone closeout.
